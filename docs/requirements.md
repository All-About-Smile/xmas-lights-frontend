## 1. 서론

### 1.1. 문서 목적

본 문서는 온라인 롤링페이퍼 웹사이트 ‘**밝혀줘! 내 X-mas 전구**’의 요구사항에 관한 명세를 정리한 SW 요구사항 명세서(Software Requirements Specification)dlau, 추후 개발할 애플리케이션의 기능 요구사항, 인터페이스 요구사항, 비기능적 요구사항에 관하여 기술한다. 이는 개발 프로세스의 진행에 있어서 아키텍처 설계와 구현 및 검증 과정에 기반이 되어 애플리케이션 요구사항의 명확한 전달을 목적으로 한다.

### 1.2. 용어 정의

#### 1.2.1 도메인 용어 정의

|   |   |
|---|---|
|용어|정의|
|**사용자 (User)**|웹사이트를 이용하는 모든 사람을 지칭. 회원 여부와 관계없이, 서비스를 탐색하거나 편지를 작성할 수 있다.|
|**회원 (Member)**|이메일과 비밀번호를 통해 계정을 생성하고 로그인한 사용자. 개인 롤링페이퍼를 관리할 수 있다.|
|**비회원 (Guest)**|계정 없이 공유 링크를 통해 롤링페이퍼 페이지에 접근하는 사용자. 편지를 작성할 수는 있으나 관리 권한은 없다.|
|**롤링페이퍼 (Rolling Paper)**|특정 사용자에게 전달되는 여러 편지들의 모음. `user_id`로 구분되며, 시각적으로는 트리 형태로 표현된다.|
|**편지 (Letter)**|롤링페이퍼를 구성하는 개별 메시지 단위. 작성자, 내용, 닉네임, 작성일 등의 속성을 가진다.|
|**편지 ID (Letter ID)**|편지마다 갖는 고유한 식별 번호. `{letter_id}`|
|**편지 주인 (Owner)**|롤링페이퍼의 주인, 즉 편지를 받는 사람. `/user_id` 경로의 페이지를 소유한다.|
|**편지 번호 (Letter Number)**|편지 주인의 편지가 갖는 순차적인 번호. `{letter_number}`|
|**오너먼트 (Ornament)**|롤링페이퍼 트리 위에 표현되는 시각적 요소로, 각각의 편지(Letter)를 상징한다. 클릭 시 해당 편지를 읽을 수 있다.|
|**편지 작성자 (Writer)**|롤링페이퍼에 새 편지를 작성하는 사람. 회원 또는 비회원 모두 가능하다.|
|**편지 리스트 페이지 (Letter List Page)**|`/user_id` 경로의 페이지. 특정 오너가 받은 모든 편지 목록(오너먼트)을 표시한다.|
|**편지 읽기 페이지 (Letter View Page)**|`/user_id/letter_id` 경로의 페이지. 선택한 편지의 내용을 표시한다.|
|**편지 작성 페이지 (Letter Write Page)**|`/user_id/write` 경로의 페이지. 닉네임을 입력하고, 편지 내용을 작성하여 제출한다.|
|**내 롤링페이퍼 (My Page)**|로그인한 사용자의 개인 페이지(`/my`). 본인이 받은 롤링페이퍼를 관리한다.|
|**공유 링크 (Share Link)**|특정 롤링페이퍼 페이지를 다른 사용자에게 전달하기 위한 URL. `/user_id` 또는 `/user_id/letter_id` 형태로 구성된다.|

|**MY**|로그인한 사용자 기능|

## 2. 개요

### **2.1 프로젝트 목적**

본 프로젝트의 목적은 **친구, 가족, 혹은 지인에게 따뜻한 메시지를 전달하고, 이를 일정 시점(12월 25일) 이후에 함께 열람할 수 있는 타임캡슐형 롤링페이퍼 웹서비스**를 개발하는 것이다. 사용자는 자신의 페이지를 생성하고, 이를 링크 형태로 공유하여 다른 사람들이 편지를 작성할 수 있도록 한다. 모아진 편지들은 크리스마스 당일 이후에만 공개되어, 기다림과 설렘을 전달하는 감성적인 경험을 제공한다.

### 2.2 핵심 컨셉

이 서비스의 핵심 컨셉은 **“크리스마스에 열리는 디지털 선물”**이다.

- 사용자가 자신의 **롤링페이퍼 링크를 공유**하면, 링크를 받은 사람들이 자유롭게 메시지를 작성한다.
    
- 각 메시지는 시각적으로 **트리 장식, 별, 스노우볼 등의 오브젝트 형태로 꾸며져** 하나의 크리스마스 오너먼트를 구성한다.
    
- **크리스마스(12월 25일)** 이후에만 편지 내용을 열람할 수 있도록 제한하여, 선물을 기다리는 듯한 **감정적 몰입감과 이벤트성**을 강화한다.
    

### **2.3 개발 범위 (Scope)**

본 프로젝트의 개발 범위는 다음과 같다:

1. **사용자 계정 관리 기능**
    
    - 회원가입, 로그인, 인증 및 개인 페이지 생성
        
2. **롤링페이퍼 관리 기능**
    
    - 개인 롤링페이퍼 페이지 생성 및 자동 URL 부여
        
    - 공유 가능한 링크 발급
        
3. **메시지 작성 기능**
    
    - 링크를 통해 접근한 작성자가 편지 작성
        
    - 오브젝트(트리 장식 등) 선택 및 시각적 배치
        
    - 비로그인 사용자 작성 허용
        
4. **메시지 조회 기능**
    
    - 2025년 12월 25일 이후 편지 열람 가능
        
    - 편지 목록 및 상세보기 인터페이스 제공
        
5. **데이터 저장 및 관리 기능**
    
    - 사용자 정보, 편지 내용, 오브젝트 선택 정보의 DB 저장 및 조회
        
    - 오류 처리 및 유효성 검증 로직 포함

## 4. 비기능적 요구사항

### **4.1 성능 요구사항 (Performance Requirements)**

- **응답 시간**
    
    - 주요 페이지(메인, 리스트, 작성, 보기)는 사용자의 요청 후 **3초 이내**에 로드되어야 한다.
        
- **동시 접속 처리**
    
    - 서버는 최소 **100명 이상의 동시 접속자**를 안정적으로 처리할 수 있어야 한다.
        
- **데이터 저장 성능**
    
    - 메시지 작성 시 DB 저장은 **1초 이내**에 완료되어야 한다.
        

### **4.2 보안 요구사항 (Security Requirements)**

- **인증 및 접근 제어**
    
    - 사용자 로그인 및 세션 관리는 **JWT 또는 세션 기반 인증** 방식을 사용한다.
        
    - 로그인하지 않은 사용자는 자신의 페이지나 타인의 데이터를 수정할 수 없다.
        
- **데이터 암호화**
    
    - 비밀번호는 **SHA-256 이상의 단방향 해시**로 암호화하여 저장한다.
        
- **통신 보안**
    
    - 모든 클라이언트-서버 통신은 **HTTPS (TLS)** 프로토콜을 사용하여 암호화한다.
        

### **4.3 사용성 요구사항 (Usability Requirements)**

- **반응형 디자인**
    
    - PC, 태블릿, 모바일 등 다양한 해상도에서 **레이아웃이 자동으로 조정**되어야 한다.
        
- **접근성**
    
    - 트리, 오너먼트 등 시각적 요소는 **색각 이상자도 인식 가능한 색 대비**를 유지해야 한다.
        
- **피드백 제공**
    
    - 버튼 클릭, 링크 복사, 오류 발생 시 사용자에게 **명확한 시각적·텍스트 피드백**을 제공해야 한다.
        

### **4.4 신뢰성 요구사항 (Reliability Requirements)**

- **오류 처리**
    
    - 서버나 네트워크 오류 발생 시 “잠시 후 다시 시도해주세요.” 등의 **안내 메시지**를 표시한다.
        
- **데이터 무결성**
    
    - 중복 데이터나 잘못된 입력이 DB에 저장되지 않도록 **유효성 검증 로직**을 구현한다.
        
- **복구 가능성**
    
    - 시스템 장애 발생 시 로그 기반으로 **데이터 복구가 가능**해야 한다.
        

### **4.5 유지보수성 요구사항 (Maintainability Requirements)**

- **코드 구조화**
    
    - 백엔드와 프론트엔드는 **명확히 분리된 계층 구조**로 구성되어야 한다.
        
    - 주요 모듈은 **API, UI 컴포넌트, 데이터 계층** 단위로 관리한다.
        
- **버전 관리**
    
    - 모든 소스 코드는 **Git**을 통해 버전 관리하며, 주요 변경사항은 커밋 메시지에 기록한다.
        
- **문서화**
    
    - 코드, API, DB 스키마에 대한 **기술 문서 및 주석**을 유지해야 한다.
        

### **4.6 확장성 요구사항 (Scalability Requirements)**

- **서비스 확장**
    
    - 향후 이벤트(예: 새해, 생일, 밸런타인데이) 버전의 롤링페이퍼로 **확장 가능**하도록 설계한다.
        
- **데이터베이스 확장성**
    
    - 사용자 수 증가에 대비해 **수평 확장(Sharding, Replication)**이 가능한 DB 구조를 고려한다.
        

### **4.7 가용성 요구사항 (Availability Requirements)**

- **서비스 가동률**
    
    - 서비스는 **연중무휴 24시간 운영**을 목표로 하며, 장애 발생 시 **1시간 이내 복구**를 목표로 한다.
        
- **백업 및 복원**
    
    - 주요 데이터는 **하루 1회 이상 자동 백업**되어야 하며, 긴급 상황 시 **즉시 복원** 가능해야 한다.
        

### **4.8 호환성 요구사항 (Compatibility Requirements)**

- **브라우저 호환성**
    
    - 최신 버전의 **Chrome, Edge, Safari, Firefox**에서 정상적으로 작동해야 한다.
        
- **웹 표준 준수**
    
    - 모든 페이지는 **HTML5, CSS3, ECMAScript 6 이상**의 웹 표준을 준수한다.
        
- **API 호환성**
    
    - 백엔드 API는 RESTful 원칙을 따르며, JSON 포맷으로 통신해야 한다.
